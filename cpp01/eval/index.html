    cpp01 評価対策¶

    このモジュールの評価テーマ¶

    メモリ管理 (new/delete)、ポインタと参照の違い、ファイル I/O、メンバ関数ポインタを理解しているか

    評価前チェック（全 exercise 共通）¶

    cpp00 と同じ前提チェック + Forbidden Function フラグが適用される。

    • c++ -Wall -Wextra -Werror -std=c++98 でコンパイルが通る
    • ヘッダに関数実装がない
    • printf / malloc / free / using namespace / friend 不使用
    • メモリリークなし (特に ex00, ex01 は delete 忘れに注意)

    Exercise 別 Q&A¶

    Ex00: BraiiiiiiinnnzzzZ¶

    質問 模範回答 なぜ聞かれるか
    Zombie クラスは private な name を持つか？ はい。_name は private で、コンストラクタでのみ設定 カプセル化の基本
    コンストラクタで name を初期化しているか？ 初期化子リスト : _name(name) を使用 正しい初期化方法を知っているか
    デストラクタにデバッグメッセージがあるか？ <name> has been destroyed を出力 オブジェクトの破棄タイミングの可視化
    announce() の出力形式は？ <name>: BraiiiiiiinnnzzzZ... 仕様通りか
    newZombie() と randomChump() の違いは？ newZombie はヒープに生成してポインタを返す。randomChump はスタックに生成して関数内で完結 スタック vs ヒープの使い分け
    ヒープの Zombie は正しく delete されているか？ main() で delete を呼んでいる。メモリリークなし メモリ管理の基本

    よく聞かれる口頭質問

    Q: なぜ newZombie はヒープに確保するのか？

    A: 関数の外でもオブジェクトを使い続けたいから。スタック上の変数は関数終了時に破棄されるため、 ポインタを返してもダングリングポインタになる。

    Q: new と malloc の違いは？

    A: new はコンストラクタを呼ぶ。malloc はメモリを確保するだけで初期化しない。 同様に delete はデストラクタを呼んでからメモリを解放するが、free は即解放。

    Ex01: Moar brainz!¶

    質問 模範回答 なぜ聞かれるか
    zombieHorde は new[] を使っているか？ はい。new Zombie[N] で配列を一括確保 配列の動的確保
    全 Zombie に名前が設定されているか？ setName() でループ内で設定 デフォルトコンストラクタ後の初期化
    delete[] で解放しているか？ はい。delete[] horde で全要素のデストラクタが呼ばれる new[] と delete[] の対応

    よく聞かれる口頭質問

    Q: delete と delete[] の違いは？

    A: delete は1 つのオブジェクトのデストラクタを呼んで解放。 delete[] は配列の全要素のデストラクタを呼んで解放。 new[] に対して delete を使うと未定義動作。

    Q: なぜデフォルトコンストラクタが必要なのか？

    A: new Zombie[N] は各要素を引数なしで構築するため。 引数付きコンストラクタしかないとコンパイルエラーになる。

    Ex02: HI THIS IS BRAIN¶

    質問 模範回答 なぜ聞かれるか
    参照のアドレスは元の変数と同じか？ はい。&stringREF == &str。参照はエイリアスだから 参照の本質を理解しているか
    参照の値は元の変数と同じか？ はい。stringREF == str 同上

    Ex03: Unnecessary violence¶

    質問 模範回答 なぜ聞かれるか
    HumanA は参照で Weapon を持っているか？ はい。Weapon &_weapon として宣言 参照 vs ポインタの使い分け
    HumanB はポインタで Weapon を持っているか？ はい。Weapon *_weapon として宣言 同上
    getType() は const std::string& を返すか？ はい。コピーを避けつつ変更を防ぐ const 参照戻り値の理解
    Weapon を変更すると attack メッセージも変わるか？ はい。両者とも同じ Weapon を参照/ポインタで保持しているから 所有権とエイリアスの理解

    最も重要な口頭質問

    Q: なぜ HumanA は参照で HumanB はポインタなのか？

    A: HumanA は常に武器を持つ設計 → 参照は NULL にならず、一度束縛したら変更不可。 HumanB は武器なしで存在できる → ポインタは NULL を許容し、後から setWeapon() で変更可能。

    判断基準: 「必ず存在 + 変更しない」→ 参照、「存在しないかもしれない or 変更する」→ ポインタ。

    Ex04: Sed is for losers¶

    質問 模範回答 なぜ聞かれるか
    std::string::replace() を使っていないか？ 使っていない。find() + substr() で手動実装 禁止関数チェック
    出力ファイル名は <input>.replace か？ はい 仕様通りか
    ファイルが開けない場合のエラー処理はあるか？ is_open() でチェックし、エラーメッセージを出力 エラー処理
    空の検索文字列は処理できるか？ 無限ループにならないよう空文字チェックを入れている エッジケース対策

    Ex05: Harl 2.0¶

    質問 模範回答 なぜ聞かれるか
    メンバ関数ポインタを使っているか？ はい。void (Harl::*funcs[4])(void) として配列を宣言 if/else 禁止の要件
    4 つのレベルが全て動作するか？ DEBUG, INFO, WARNING, ERROR それぞれでメッセージが出力される 基本動作確認

    よく聞かれる口頭質問

    Q: メンバ関数ポインタの構文を説明して

    A: 宣言: void (Harl::*func)(void) — Harl クラスのメンバ関数へのポインタ。 代入: func = &Harl::debug — & とクラス名が必須。 呼び出し: (this->*func)() — ->* 演算子でオブジェクト経由で呼ぶ。

    Q: なぜ通常の関数ポインタではダメなのか？

    A: メンバ関数は暗黙の this 引数を持つため、通常の関数ポインタとは型が異なる。 void (*)(void) と void (Harl::*)(void) は別の型。

    Ex06: Harl filter¶

    質問 模範回答 なぜ聞かれるか
    switch 文を使っているか？ はい 要件の確認
    fall-through で指定レベル以上が表示されるか？ はい。break を意図的に省略して、以降の case も実行される fall-through の理解
    不正なレベルで適切なメッセージが出るか？ default で "insignificant problems" メッセージ エラー処理

    よく聞かれる口頭質問

    Q: fall-through とは何か？

    A: switch 文で break を書かないと、次の case の処理も続けて実行されること。 通常はバグだが、ここでは「WARNING なら WARNING + ERROR を表示」するために意図的に利用している。

    概念の深掘り¶

    スタック vs ヒープ¶

    ┌─────────────── プログラムメモリ ───────────────┐
│                                                │
│  スタック (Stack)                 ヒープ (Heap)  │
│  ┌──────────────┐               ┌────────────┐ │
│  │ main()       │               │            │ │
│  │  Zombie z;   │  ← 自動破棄   │ new Zombie │ │
│  │  int x;      │               │            │ │
│  ├──────────────┤               │ → delete   │ │
│  │ func()       │               │   するまで  │ │
│  │  Zombie z2;  │  ← func終了で │   生き続ける│ │
│  │              │     自動破棄   │            │ │
│  └──────────────┘               └────────────┘ │
│  高速 / サイズ制限あり          遅い / 大きい    │
│  管理不要                       管理が必要       │
└────────────────────────────────────────────────┘

    ポインタ vs 参照の判断フロー¶

    対象が NULL (不在) になる可能性がある？
  ├── YES → ポインタ (Weapon *_weapon)
  │         └── 後から付け替える可能性がある？
  │             ├── YES → ポインタ + setter
  │             └── NO  → ポインタ (ただし初期化後は変更しない)
  └── NO → 参照 (Weapon &_weapon)
           └── 必ず初期化子リストで初期化する

    即不合格フラグ一覧¶

    フラグ 条件
    Forbidden function malloc/free/printf/using namespace/friend
    Leaks new に対応する delete がない、new[] に delete (非配列) を使用
    Crash NULL ポインタのデリファレンス、ダングリングポインタ
    Can't support / explain code メンバ関数ポインタの構文を説明できない、参照 vs ポインタの判断基準を答えられない
