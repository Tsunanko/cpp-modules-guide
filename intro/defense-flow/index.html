    ディフェンス（評価）の流れ¶

    このページは何？¶

    42 のピアレビュー（ディフェンス）がどう進むかを解説するページ です。

    「ディフェンスって何をされるの？」 「どんな質問が来るの？」 「何を準備しておけばいい？」

    この疑問に答えます。

    ディフェンスって何？¶

    他の学生（ピア）に自分のコードを見せて評価してもらうイベント です。

    42 の課題は提出するだけでは合格にならず、 3 人のピアに評価してもらう必要があります。

    提出
  ↓
vogsphere に push
  ↓
ピアがランダムに割り当てられる
  ↓
対面 or リモートで 30 分程度の評価
  ↓
3 人合格 → ポイント獲得 → 課題合格

    全体の流れ（30 分）¶

    フェーズ 1: 準備確認（5 分）¶

    レビュアーがチェックするもの:

    1. Git リポジトリが正しいか
      • git clone が空のフォルダにできるか
      • 正しいプロジェクトか
    2. 不正がないか
      • alias でごまかしていないか
      • 余計なスクリプトが動いていないか

    準備しておくもの

    • git status をクリーンにしておく
    • .gitignore で .o やバイナリを除外
    • 評価用のフォルダを別で用意

    フェーズ 2: コンパイル（3 分）¶

    make
→ 警告もエラーもなく通るか？

    ここで失敗 = 即 0 点

    • -Wall -Wextra -Werror のフラグが必要
    • 警告も 0 にすること（-Werror は警告をエラー扱い）
    • make re で再ビルドも確認される

    フェーズ 3: 動作確認（10 分）¶

    各 exercise について:

    1. subject の期待動作が再現できるか
    2. エッジケース（空入力、EOF 等）を試す
    3. 不正な入力でクラッシュしないか

    レビュアーは subject の通りに試す

    subject の実行例を必ず試されます。 提出前に自分でも subject 通りに動くか確認しましょう。

    フェーズ 4: コードレビュー（10 分）¶

    ソースコードを見ながら口頭質問されます。

    よく聞かれる質問の種類:

    種類 例
    仕様 「この関数は何をしている？」
    設計 「なぜ private にした？」
    C++ 知識 「std::endl と "\n" の違いは？」
    メモリ 「new した後の delete はどこ？」

    フェーズ 5: 評価確定（2 分）¶

    レビュアーが採点シートを記入:

    • Yes / No で評価する項目
    • 0-5 点 で評価する項目
    • フラグ（Crash, Leaks, Forbidden function 等）

    即 0 点になるフラグ一覧¶

    絶対避けるフラグ

    フラグ 意味
    Invalid compilation コンパイルエラー or 警告
    Empty work 提出物が空
    Incomplete work 未完成
    Crash segfault やクラッシュ
    Leaks メモリリーク
    Forbidden function 禁止関数（malloc, printf, using namespace 等）
    Cheat 不正行為
    Can't support / explain code 自分のコードを説明できない
    Concerning situation 倫理的に問題あり

    特に注意すべきフラグ¶

    Forbidden function¶

    // ❌ 禁止
#include <cstdio>
printf("...");
malloc(...);
using namespace std;

// ✅ OK
std::cout << "...";
new Type;
std::cout << "..." << std::endl;

    Leaks¶

    // ❌ メモリリーク
int *arr = new int[10];
// delete[] を忘れて関数終了

// ✅ 必ず対にする
int *arr = new int[10];
// ... 使う ...
delete[] arr;

    macOS なら leaks コマンド、 Linux なら valgrind で確認できます。

    # macOS
leaks --atExit -- ./my_program

# Linux
valgrind --leak-check=full ./my_program

    Can't support / explain code¶

    コードが動いていても、説明できないと 0 点になります。

    特によく聞かれる説明:

    • 「なぜ const を付けた？」
    • 「なぜ参照を使った？ ポインタじゃダメ？」
    • 「この virtual はどういう意味？」
    • 「コンストラクタの呼ばれる順番は？」

    ディフェンス前のチェックリスト¶

    提出前の最終確認¶

    • make が警告なしで通る
    • make re もちゃんと動く
    • make fclean で全部消える
    • メモリリークなし（leaks で確認）
    • 禁止関数を使っていない
    • subject の実行例が全部動く
    • 各 exercise のチェックリストを確認した
    • 評価対策ページの Q&A を見直した

    当日持参するもの¶

    • 自分のコード（Git リポジトリにある）
    • ラップトップ（動作確認用）
    • 自分用のメモ（よく質問されることを書いておく）

    よく聞かれる質問トップ 10（全モジュール共通）¶

    1. 「なぜ std::endl を使う？ "\n" ではダメ？」¶

    答え: endl は改行 + バッファのフラッシュ。 "\n" は改行のみ。大量出力では "\n" の方が速い。

    2. 「using namespace std; はなぜ禁止？」¶

    答え: 名前の衝突を招くから。 大規模プロジェクトでは危険。

    3. 「なぜメンバ変数は private ？」¶

    答え: カプセル化。外部から勝手に変更されないようにする。 バリデーションを setter に入れられる。

    4. 「const を付ける理由は？」¶

    答え: 「この関数/変数は変更しない」という約束。 コンパイラが強制してくれるのでバグを防げる。

    5. 「new と malloc の違いは？」¶

    答え: new はコンストラクタを呼ぶ。 malloc はメモリを確保するだけ。

    6. 「参照とポインタの使い分けは？」¶

    答え:

    • 必ず存在 + 変更しない → 参照
    • NULL の可能性 or 変更する → ポインタ

    7. 「コンストラクタの初期化子リストって？」¶

    答え: : _name(name) の形。 コンストラクタ本体に入る前に初期化する。 const メンバや参照メンバには必須。

    8. 「継承の構築/破棄順序は？」¶

    答え:

    • 構築: 親 → 子（LIFO のスタック）
    • 破棄: 子 → 親（逆順）

    9. 「virtual 関数って何？」¶

    答え: 実行時に「本当の型」の関数を呼ぶ仕組み。 ポリモーフィズムの基礎。

    10. 「ディープコピーとシャローコピーの違いは？」¶

    答え:

    • シャローコピー: ポインタをそのままコピー（同じ中身を指す）
    • ディープコピー: 指している先もコピー（独立した 2 つ）

    メンバにポインタがある場合はディープコピーが必須。

    心構え¶

    ディフェンスは「自分を試す場」

    レビュアーは敵ではなく、同じ学生です。 分からないことは正直に「分かりません」と言ってOK。

    ただし、自分で書いたコードは説明できるように。 コピペではなく、自分の言葉で説明できることが大事です。

    相手にも敬意を

    自分が評価する側の時も、建設的に評価しましょう。 評価システム（EVP）は全員で支える仕組みです。
